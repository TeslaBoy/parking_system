2 Спеціальна частина
2.1 Розробка інформаційної системи управління паркінгом

2.1.1 Постановка задачі

Інформаційна система управління паркінгом призначена для автоматизації процесів бронювання, контролю та обліку парковочних місць. 
Система забезпечує функціональність для двох категорій користувачів: звичайних користувачів та адміністраторів. 
Звичайні користувачі мають можливість реєструватися, авторизовуватися, додавати транспортні засоби з реєстраційними номерами, переглядати доступні паркінги з індикацією зайнятості, бронювати місця на певний час, а також керувати своїми бронюваннями (перегляд, скасування). 
Адміністратори отримують розширений функціонал: управління паркінгами (додавання, редагування, видалення), моніторинг транспорту, перегляд та обробку всіх бронювань з можливістю підтвердження або відхилення, а також доступ до аналітичних даних.

2.1.2 Мета створення інформаційної системи

Метою створення системи є забезпечення ефективного управління парковочними місцями шляхом автоматизації ключових бізнес-процесів. Система повинна відповідати на такі питання:
- яка загальна кількість парковочних місць доступна на кожному паркінгу;
- скільки місць вже зайнято та скільки залишилося вільних;
- які транспортні засоби належать конкретному користувачу та де вони розташовані;
- які бронювання були здійснені користувачем, їх статус та терміни;
- які паркінги доступні для бронювання, їх розташування та вартість;
- які бронювання очікують підтвердження від адміністратора;
- загальний дохід системи та завантаженість паркінгів.
Додатковою метою є створення зручного веб-інтерфейсу, що забезпечує швидке виконання операцій без необхідності оновлення сторінки.

2.1.3 Аналіз предметної області

В рамках предметної області управління паркінгом виділяються наступні базові сутності:

1) Користувачі — особи, які мають доступ до системи. Кожен користувач має унікальний обліковий запис з логіном, хешованим паролем, електронною адресою та роллю (адміністратор або звичайний користувач). Користувачі з роллю «адміністратор» мають повний доступ до всіх об'єктів системи, тоді як звичайні користувачі можуть керувати лише власними даними. Один користувач може мати кілька транспортних засобів.

2) Транспортні засоби — об'єкти, які належать користувачам. Кожний транспортний засіб характеризується реєстраційним номером, маркою, моделлю, кольором, а також має можливість зберігати зображення. Транспортний засіб може бути призначений на певний паркінг (вказує місце поточного розташування) або перебувати в режимі «Не припарковано».

3) Паркінги — об'єкти, що представляють парковочні зони. Кожен паркінг має назву, адресу, загальну місткість (кількість паркомісць), кількість доступних місць, тариф за годину та зображення. Система автоматично відстежує зміну кількості доступних місць при створенні, підтвердженні, скасуванні або завершенні бронювання.

4) Бронювання — процес резервації парковочного місця на певний час. Кожне бронювання зв'язує користувача, транспортний засіб та паркінг, містить часові межі (початок і кінець), розраховану суму, статус та дату створення. 
Статуси бронювання: «pending» (очікує підтвердження адміністратором), «active» (активне — місце зайнято), «completed» (завершене — час вийшов), «cancelled» (скасоване). 
Підтвердження бронювання адміністратором змінює статус з «pending» на «active»; при скасуванні або відхиленні паркомісце автоматично повертається у паркінг.

2.1.4 Проєктування та створення бази даних

Згідно аналізу предметної області, база даних містить чотири основні таблиці з визначеними зв'язками. 
Структура бази даних реалізована за допомогою MySQL — активної реляційної системи керування базами даних, що забезпечує швидкість, надійність та підтримку транзакцій.

Таблиця users — зберігає інформацію про користувачів системи:
CREATE TABLE IF NOT EXISTS users (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(30) NOT NULL,
    password VARCHAR(255) NOT NULL,
    email VARCHAR(50),
    role ENUM('admin', 'user') DEFAULT 'user',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
Таблиця містить унікальний ідентифікатор, логін, хеш пароля (з використанням алгоритму PASSWORD_DEFAULT), електронну адресу, роль користувача та дату створення облікового запису. 
При створенні бази даних автоматично додаються тестові користувачі: адміністратор (логін: admin, пароль: admin123) та звичайний користувач (логін: user, пароль: user123).

Таблиця parking_places — зберігає інформацію про паркінги:
CREATE TABLE IF NOT EXISTS parking_places (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    address VARCHAR(200) NOT NULL,
    capacity INT(4) NOT NULL,
    available INT(4) NOT NULL,
    price_per_hour DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
    image VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
Таблиця містить унікальний ідентифікатор, назву паркінгу, адресу, загальну місткість, кількість доступних місць, тариф за годину та поле для зображення. 
При створенні бази даних автоматично додаються три тестові паркінги у місті Кам'янське: Центральний паркінг (вул. Будівельників, 1, тариф 20 грн/год), Парковка біля ЦУМу (пр. Свободи, 35, тариф 30 грн/год) та Паркінг Залізничний (вул. Залізнична, 10, тариф 15 грн/год).

Таблиця vehicles — зберігає інформацію про транспортні засоби:
CREATE TABLE IF NOT EXISTS vehicles (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    license_plate VARCHAR(20) NOT NULL,
    brand VARCHAR(50) NOT NULL,
    model VARCHAR(50) NOT NULL,
    color VARCHAR(30) NOT NULL,
    user_id INT(6) UNSIGNED,
    parking_id INT(6) UNSIGNED,
    image VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (parking_id) REFERENCES parking_places(id)
)
Таблиця містить унікальний ідентифікатор, реєстраційний номер, марку, модель, колір, а також зовнішні ключі до користувача (власника) та паркінгу (місце розташування). 
При створенні бази даних для тестового користувача автоматично додаються два транспортні засоби: Tesla Model 3 (номера AE 1234 BT, колір червоний) та BMW X5 (номера AE 5678 CP, колір чорний).

Таблиця bookings — зберігає інформацію про бронювання:
CREATE TABLE IF NOT EXISTS bookings (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id INT(6) UNSIGNED,
    parking_id INT(6) UNSIGNED,
    vehicle_id INT(6) UNSIGNED,
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    total_price DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
    status ENUM('pending', 'active', 'completed', 'cancelled') DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (parking_id) REFERENCES parking_places(id) ON DELETE CASCADE,
    FOREIGN KEY (vehicle_id) REFERENCES vehicles(id) ON DELETE CASCADE
)
Таблиця містить унікальний ідентифікатор, зовнішні ключі до користувача, паркінгу та транспортного засобу, часові межі бронювання, розраховану суму, статус та дату створення. 
При видаленні бронювання автоматично повертається паркомісце (через ON DELETE CASCADE та оновлення available). 
Структуру взаємовідносин між таблицями бази даних наведено на рисунку 2.1.

2.1.5 Написання SQL запитів до бази даних

Система використовує набір SQL-запитів для забезпечення функціонування та взаємодії з базою даних, що дозволяє реалізувати ключові операції користувачів та адміністратора. 
Всі запити до бази даних виконуються через підготовлені вирази (prepared statements) за допомогою MySQLi для захисту від SQL-ін'єкцій.

Для отримання повного переліку доступних паркінгів використовується простий запит:
SELECT * FROM parking_places ORDER BY id DESC

Для отримання списку транспортних засобів конкретного користувача застосовується об'єднання таблиць vehicles та parking_places:
SELECT v.*, p.name as parking_name FROM vehicles v 
LEFT JOIN parking_places p ON v.parking_id = p.id 
WHERE v.user_id = ? ORDER BY v.id DESC

Запит для отримання бронювань користувача здійснює об'єднання чотирьох таблиць (bookings, users, parking_places, vehicles), що дозволяє отримати повну інформацію:
SELECT b.*, u.username, p.name as parking_name, p.address as parking_address, 
       v.license_plate, v.brand, v.model, v.color 
FROM bookings b 
JOIN users u ON b.user_id = u.id 
JOIN parking_places p ON b.parking_id = p.id 
JOIN vehicles v ON b.vehicle_id = v.id 
WHERE b.user_id = ? ORDER BY b.start_time DESC

Додавання нового паркінгу реалізується за допомогою підготовленого запиту з підтримкою завантаження зображення:
INSERT INTO parking_places (name, address, capacity, available, price_per_hour, image) 
VALUES (?, ?, ?, ?, ?, ?)

Реєстрація нового транспортного засобу:
INSERT INTO vehicles (license_plate, brand, model, color, parking_id, image, user_id) 
VALUES (?, ?, ?, ?, ?, ?, ?)

Створення нового бронювання виконується запитом з automatic встановленням статусу «pending» та розрахунком суми на основі тарифу паркінгу:
INSERT INTO bookings (user_id, parking_id, vehicle_id, start_time, end_time, total_price, status) 
VALUES (?, ?, ?, ?, ?, ?, 'pending')

Для підтримки актуальної інформації про кількість доступних місць при створенні бронювання застосовується оновлення:
UPDATE parking_places SET available = available - 1 WHERE id = ?

При підтвердженні бронювання статус змінюється:
UPDATE bookings SET status = 'active' WHERE id = ?

При скасуванні або відхиленні бронювання місце повертається:
UPDATE parking_places SET available = available + 1 WHERE id = ?

Автоматичне завершення прострочених бронювань реалізовано через псевдо-Cron на кожному завантаженні сторінки:
UPDATE parking_places SET available = available + 1 WHERE id = ?
UPDATE bookings SET status = 'completed' WHERE id = ?

2.2 Розробка інтерфейсу користувача

2.2.1 Планування інтерфейсу та розробка макету

Інтерфейс системи розроблено з використанням сучасної технології Bootstrap 5.3 для забезпечення адаптивності, сумісності з мобільними пристроями та професійного візуального вигляду. 
Архітектура інтерфейсу будується на принципах Single Page Application (SPA): навігація між розділами відбувається без повного перезавантаження сторінки за допомогою Bootstrap-табів та AJAX-запитів.
Система має дві відповідальні частини інтерфейсу: адміністративну панель та користувацький кабінет, кожна з яких адаптована під конкретні завдання відповідного рівня доступу.

2.2.2 Структура інтерфейсу та навігація

Центральним елементом інтерфейсу є бічне навігаційне меню (сайдбар), закріплене на лівій стороні екрану. 
Сайдбар містить: логотип системи, блок навігаційних посилань, розділених на групи залежно від ролі користувача (адміністрування для адміністраторів, навігація для звичайних користувачів), а також футер з інформацією про поточного користувача (аватар, логін, роль) та кнопкою виходу. 
На мобільних пристроях сайдбар автоматично переміщується за межі екрану, замість нього з'являється верхня панель (topbar) з кнопкою гамбургер-меню та написом системи.

Для адміністраторів навігаційне меню містить чотири розділи:
- Дашборд — загальна статистика, KPI-показники, графік завантаженості, швидкий доступ до останніх бронювань та паркінгів;
- Паркінги — повний список паркінгів із можливістю додавання, редагування та видалення;
- Транспорт — перегляд усіх транспортних засобів в системі;
- Бронювання — всі бронювання з можливістю підтвердження, відхилення, редагування та видалення.

Для звичайних користувачів навігація включає:
-Паркінги — картки доступних паркінгів із завантаженістю;
- Мої бронювання — історія та поточні бронювання;
- Мій транспорт — перелік особистих транспортних засобів.

2.2.3 Адміністративна панель

Адміністративна панель відкривається на вкладці «Дашборд» та складається з наступних блоків:

а) Заголовок сторінки з назвою «Панель адміністратора», підзаголовком «Огляд системи управління паркінгом» та кнопкою експорту звіту у форматі CSV.

б) Блок KPI-карток (чотири картки в ряд):
- «Загальний дохід» — сума всіх завершених та активних бронювань;
- «Активні паркування» — кількість автомобілів, що зараз знаходяться на стоянках;
- «Всього транспорту» — загальна кількість транспортних засобів в системі;
- «Заповненість» — відсоток зайнятих місць з підказкою вільних/загальних.
Кожна картка має кольорову індикацію (зелений, синій, блакитний, жовтий), іконку, заголовок, значення та допоміжний текст. При наведенні картка піднімається вгору з тіню.

в) Графік завантаженості паркінгів — стовпчаста діаграма (Chart.js), що відображає кількість зайнятих місць по кожному паркінгу. Кольори стовпців динамічно змінюються залежно від рівня заповнення: зелений (< 50%), жовтий (50–90%), червоний (> 90%).

г) Блок «Статистика сьогодні» — чотири показники: нових бронювань за сьогодні, активних, паркінгів в системі, вільних місць.

д) Таблиця «Управління паркінгами» — список паркінгів з мініатюрою зображення, назвою, адресою, тарифом, кількістю місць, прогрес-баром заповнення та кнопками редагування/видалення.

е) Блок «Останні бронювання» — таблиця з 10 останніми бронюваннями, користувачем, транспортом, паркінгом, часом, сумою, статусом та діями (підтвердити/відхилити для pending, редагувати/видалити для інших).

є) Блок «Усі транспортні засоби» — таблиця з переліком транспорту, місцем розташування, діями редагування/видалення.

2.2.4 Користувацький кабінет

Користувацький інтерфейс організовано навколо вітального блоку — градієнтної смужки з привітанням, підзаголовком та кнопкою додавання транспорту. 
Нижче розташовані три вкладки:

«Доступні паркінги» — картки паркінгів у сітці (3 колонки на десктопі, 2 на планшеті, 1 на мобільному). Кожна картка містить: зображення паркінгу (або плейсхолдер), бейдж з кількістю вільних місць (кольоровим кодуванням), назву, адресу, тариф, прогрес-бар зайнятості та кнопку «Забронювати» (або «Немає місць» для повних паркінгів). 
При натисканні кнопки «Забронювати» відкривається модальне вікно з вибором транспортного засобу, вказівкою часу заїзду/виїзду та динамічним розрахунком вартості.

«Мої бронювання» — таблиця з історією бронювань з показом паркінгу, адреси, транспорту, періоду, суми, статусу (кольоровими бейджами: зелений — активне, жовтий — очікує, сірий — завершено, червоний — скасовано) та кнопкою скасування для активних/очікуючих бронювань.

«Мій транспорт» — таблиця з переліком особистих транспортних засобів: фото (або плейсхолдер), марка/модель, реєстраційний номер, колір, поточна локація, кнопки редагування/видалення.

2.2.5 Модальні вікна

Всі операції створення та редагування об'єктів реалізовані через модальні вікна Bootstrap з темними, синім, жовтим градієнтними заголовками залежно від типу дії. 
Кожне модальне вікно містить:
- Заголовок з іконкою та назвою дії;
- Форму з кастомними полями вводу з лейблами та іконками Bootstrap Icons (наприклад, bi-building, bi-geo-alt, bi-currency-dollar);
- Спеціалізовані секції: для бронювання — картка з назвою паркінгу та тарифом; для паркінгів — три колонки (тариф, місткість, доступно); для фото — зона завантаження файлів з dashed border;
- Кнопки «Скасувати» та основний action-клас (Додати/Зберегти зміни/Оновити) на всю ширину футера.
Модальне вікно бронювання містить також динамічний блок «Орієнтивна сума» з автоматичним перерахунком при зміні часових проміжків.

2.2.6 Сторінки авторизації та реєстрації

Сторінка входу (login.php) відкривається при відсутності активної сесії. Її інтерфейс побудовано на повноекранному градієнтному фоні (від primary до dark) з декоративними колами. 
Центральна картка містить: логотип системи, заголовок «Вхід до системи», підзаголовок, форму з іконованими полями (користувач/пароль), кнопку входу та посилання на модальне вікно реєстрації. 
Модальне вікно реєстрації використовує ті самі кастомні компоненти форми, що й інші модалки системи.

2.3 Розробка серверної частини

2.3.1 Авторизація та управління сесіями

Авторизація реалізована в login.php з використанням сесій PHP. При успішній перевірці логіна та пароля (password_verify з PASSWORD_DEFAULT) створюється сесія з збереженням user_id, username та role. 
При невдалій спробі відображається повідомлення про помилку (неправильний пароль або користувача не знайдено).
Скрипт logout.php знищує сесію та перенаправляє на сторінку входу.

2.3.2 Центральний API (api.php)

Всі операції з даними виконуються через єдиний API-скрипт api.php, що обробляє GET/POST-запити з параметром action. 
Архітектура побудована за принципом RESTful API: кожна дія (додавання, оновлення, видалення, отримання, зміна статусу, експорт) викликається окремим запитом з відповідним action.
Скрипт забезпечує:
- Перевірку авторизації для всіх дій (крім register);
- Перевірку ролі адміністратора для обмежених операцій;
- Валідацію вхідних даних;
- Використання транзакцій ($conn->begin_transaction()) для атомарних операцій;
- Управління зображеннями через клас Uploader;
- Вивантаження звіту у форматі CSV з BOM для коректного відображення кирилиці в Excel.

Основні дії API:
- add_parking / update_parking / delete_parking — CRUD для паркінгів;
- add_vehicle / update_vehicle / delete_vehicle — CRUD для транспортних засобів;
- add_booking / update_booking / delete_booking / change_booking_status — CRUD та зміна статусу бронювань;
- get_parking / get_vehicle / get_booking — отримання даних одного об'єкта для модальних вікон;
- export_bookings — експорт всіх бронювань у CSV;
- register — реєстрація нового користувача.

2.3.3 Модальні вікна та обробка форм

Модальні вікна (modals.php) відповідають за збір даних від користувача. Кожне вікно прив'язане до конкретної сутності (паркінг, транспорт, бронювання) та містить форму з відповідними полями. 
Під час відправки форми дані передаються через FormData (з підтримкою файлів) на відповідний endpoint API, результат обробляється асинхронно (AJAX). 
При успішному виконанні: модальне вікно закривається, форма очищається, відображається повідомлення про успіх (toast-сповіщення SweetAlert2), інтерфейс оновлюється без перезавантаження сторінки. 
При помилці відображається повідомлення з деталем.

2.3.4 Клієнтська логіка та інтерактивність

Файл js/script.js реалізує весь клієнтський функціонал системи:
- Ініціалізація DataTables для всіх таблиць з українською локалізацією;
- Ініціалізація Chart.js для візуалізації завантаженості паркінгів;
- Обробка подій сайдбару (toggle для мобільних, перемикання табів);
- Обробка подій для всіх кнопок (редагування, видалення, підтвердження бронювань) через event delegation на document — забезпечує роботу навіть після динамічного оновлення DOM;
- Динамічний розрахунок вартості бронювання при зміні часових меж;
- Функція updateUI() — оновлює вміст табів без перезавантаження сторінки, зберігаючи активну вкладку;
- Функція reinitForms() — переініціалізує обробники форм після кожного оновлення DOM, що гарантує функціональність кнопок.

2.3.5 Сервісна логіка

Для підтримки актуальності даних реалізовано автоматичне завершення прострочених бронювань. При кожному завантаженні сторінки виконується запит, що знаходить активні бронювання з end_time < NOW() та переводить їх у статус «completed», одночасно повертаючи паркомісце в паркінг. 
Це забезпечує актуальність інформації про вільні місця без необхідності в окремому планувальнику завдань.

2.4 Результат роботи програми

2.4.1 Загальний опис інтерфейсу

Система забезпечує сучасний, адаптивний інтерфейс, оптимізований для роботи на десктопних та мобільних пристроях. 
При відкритті системи неавторизований користувач бачить сторінку входу з градієнтним фоном та карткою авторизації. Після успішного входу користувач попадає до особистого кабінету.

2.4.3 Адміністративна панель

Адміністратор бачить головний дашборд (рис. 2.1) з чотирма KPI-картками, графіком завантаженості, блоком сьогоденняї статистики, а також міні-таблицями паркінгів, бронювань та транспорту. 
Ширина смуги прогресу заповнення паркінгу відповідає відсотковому співвідношенню зайнятих місць, а кольорове кодування підвищує наочність: зелений — зайнятість менше 50%, жовтий — 50–90%, червоний — більше 90%.

Перехід між розділами (Дашборд, Паркінги, Транспорт, Бронювання) відбувається через клік на відповідний пункт сайдбару, що відповідає Bootstrap-табам. Активний розділ підсвічується в сайдбарі, що забезпечує чітку орієнтацію користувача.

На вкладці «Паркінги» (рис. 2.2) адміністратор має повний список паркінгів із зображеннями, назвою, адресою, тарифом, кількістю місць, прогрес-баром та кнопками редагування/видалення. Кнопка «Додати паркінг» відкриває модальне вікно з формою, що містить іконні поля для назви, адреси, тарифу, місткості, доступних місць та завантаження зображення.

На вкладці «Транспорт» (рис. 2.3) відображаються всі транспортні засоби з фото (або іконкою-плейсхолдером), маркою/моделлю, реєстраційним номером, місцем розташування. Можливі дії: редагування (відкривається модальне вікно з попередньо заповненими даними) та видалення (з підтвердженням через SweetAlert2).

На вкладці «Бронювання» (рис. 2.4) адміністратор бачить усі бронювання в системі. Для записів зі статусом «очікує» доступні кнопки підтвердження (зелена, bi-check-lg) та відхилення (червона, bi-x-lg). Для інших статусів — редагування та видалення. Кнопка «Експорт звіту» дозволяє завантажити CSV-файл з усіма бронюваннями.

2.4.5 Користувацький кабінет

Звичайний користувач бачить привітальний блок з ім'ям та кнопкою додавання транспорту. Нижче — вкладка «Доступні паркінги» з картками (рис. 2.5). Кожна картка містить зображення з ефектом збільшення при наведенні, бейдж вільних місць, назву, адресу, тариф, прогрес-бар та кнопку бронювання. Кнопка «Забронювати» відкриває модальне вікно з попередньо заповненим паркінгом, де користувач обирає транспорт, вказує час, а система автоматично розраховує вартість.

Вкладка «Мої бронювання» (рис. 2.6) містить таблицю з історією бронювань. Статуси відображаються кольоровими бейджами. Для активних або очікуючих бронювань доступна кнопка «Скасувати».

Вкладка «Мій транспорт» (рис. 2.7) дозволяє керувати особистими транспортними засобами: додавати (через модальне вікно з полями номера, марки, моделі, кольору, фото), редагувати та видаляти.

2.4.1 Функціональне тестування системи

Тестування системи проводилось з метою перевірки коректності роботи всіх основних функціональних блоків, відповідності вимогам та виявлення можливих дефектів. Усі операції перевірялись на різних ролях користувачів (адміністратор/звичайний користувач) та в різних браузерах (Google Chrome, Mozilla Firefox, Microsoft Edge).

А. Тестування процесу авторизації та реєстрації

При першому відкритті системи неавторизованому користувачу відкривається сторінка входу (рис. 2.8). Інтерфейс містить форму з полями «Логін» та «Пароль». 
При введенні коректних облікових даних (адміністратор: admin / admin123, користувач: user / user123) відбувається успішна авторизація з перенаправленням на головну сторінку. 
При введенні неіснуючого логіна система відображає помилку «Користувача не знайдено». При введенні неправильного пароля — «Неправильний пароль».

Якщо у користувача немає облікового запису, він може натиснути посилання «Зареєструватися», що відкриває модальне вікно реєстрації (рис. 2.9). 
Форма реєстрації містить поля: логін, електронна адреса, пароль, підтвердження пароля. При спробі зареєструватися з уже існуючим логіном система виводить помилку «Користувач вже існує». 
При успішній реєстрації обліковий запис створюється з роллю «користувач» за замовчуванням.

Вихід із системи здійснюється через кнопку в футері сайдбару — скрипт logout.php знищує сесію та перенаправляє на сторінку входу.

Б. Тестування адміністративної панелі (роль «admin»)

Б.1. Перегляд дашборду

Після входу під обліковим записом адміністратора відкривається дашборд (рис. 2.10). 
Відображаються чотири KPI-картки: «Загальний дохід» (розраховано як сума total_price всіх активних та завершених бронювань), «Активні паркування» (кількість бронювань зі статусом active), «Всього транспорту» (загальна кількість записів у таблиці vehicles), «Заповненість» (відсоток зайнятих місць).

Нижче розташовано стовпчасту діаграму, побудовану за даними з таблиці parking_places. Вісь Y позначає кількість зайнятих місць (capacity − available), вісь X — назви паркінгів. 
Стовпці зчитуються з масиву chartLabels (назви) та chartData (значення), а їхній фон визначається функцією, яка повертає '#22c55e' при заповненні менше 50%, '#eab308' при 50–90% та '#ef4444' при більше 90%.

Далі розташований блок «Статистика сьогодні» з показниками: нових бронювань за поточну дату (WHERE DATE(created_at) = CURDATE()), активних бронювань, загальної кількості паркінгів, вільних місць (SUM(available) FROM parking_places).

Б.2. Управління паркінгами

Перехід до вкладки «Паркінги» відбувається через сайдбар (рис. 2.11). 
Відображається таблиця з усіма паркінгами: мініатюра зображення, назва, адреса, тариф (грн/год), місткість/доступно, прогрес-бар (ширина = (capacity − available) / capacity × 100), кнопки редагування та видалення. 
Кнопка «Додати паркінг» відкриває модальне вікно (рис. 2.12). При заповненні форми з іконованими полями (bi-building для назви, bi-geo-alt для адреси, bi-currency-dollar для тарифу) та завантаженні зображення (upload-area з dashed border) система створює запис у БД через INSERT-запит. 
Після успішного додавання модальне вікно закривається, відображається toast-сповіщення, таблиця оновлюється AJAX-запитом.

Редагування паркінгу: при натисканні кнопки редагування (bi-pencil-square) виконується AJAX-запит get_parking&id=X, дані попередньо заповнюють форму модального вікна. Після зміни та збереження виконується UPDATE-запит, зображення може бути оновлено через клас Uploader. 
Видалення паркінгу: при натисканні кнопки видалення (bi-trash) з'являється підтверджувальне вікно SweetAlert2. При підтвердженні виконується DELETE-запит, паркомісце повертається до системи.

Б.3. Управління транспортними засобами

Вкладка «Транспорт» (рис. 2.13) містить таблицю з усіма транспортними засобами системи: фото (або іконка-плейсхолдер якщо фото відсутнє), марка, модель, реєстраційний номер, колір, поточна локація (назва паркінгу або «Не припарковано»). 
Дії: редагування (відкриває модальне вікно з попередньо заповненими даними — номер, марка, модель, колір, паркінг) та видалення (з підтвердженням). 
Додавання нового транспортного засобу через модальне вікно з полями: реєстраційний номер, марка (наприклад, Tesla, BMW, Audi), модель (наприклад, Model 3, X5), колір (color-picker), паркінг (випадаючий список з усіма паркінгами), фото.

Б.4. Обробка бронювань

Вкладка «Бронювання» (рис. 2.14) показує всі бронювання системи з сортуванням за часом створення. 
Для статусу «очікує» (pending) відображаються дві кнопки: підтвердити (зелена, bi-check-circle) та відхилити (червона, bi-x-circle). 
При підтвердженні виконується UPDATE bookings SET status = 'active' WHERE id = X — бронювання стає активним, місце фіксується за потрібним автомобілем. 
При відхиленні або скасуванні виконується UPDATE parking_places SET available = available + 1 WHERE id = X — паркомісце повертається в загальний доступ. 
Кнопка «Редагувати» (bi-pencil) відкриває модальне вікно, де можна змінити час, транспорт, паркінг, статус. 
Кнопка «Експорт звіту» генерує CSV-файл з усіма бронюваннями, з заголовками на кирилиці та BOM-маркером для коректного відкриття в Microsoft Excel.

В. Тестування користувацького кабінету (роль «user»)

В.1. Перегляд доступних паркінгів

Після входу під звичайним обліковим записом користувач бачить вітальний блок з ім'ям та кнопкою «Додати транспорт». 
Нижче — вкладка «Доступні паркінги» (рис. 2.15) з картками у сітці. Кожна картка містить зображення, бейдж з кількістю вільних місць (кольоровим кодуванням), назву, адресу, тариф, прогрес-бар зайнятості.

При натисканні кнопки «Забронювати» відкривається модальне вікно (рис. 2.16) з попередньо заповненим паркінгом (ім'я та тариф відображаються в картці всередині модалки). 
Користувач обирає транспортний засіб з випадаючого списку, вказує час заїзду (datetime-local, мінімальне значення — поточна дата/час) та час виїзду. 
При зміні часу система автоматично розраховує вартість: diffHours = Math.ceil((end − start) / (1000×60×60)), estimatedPrice = diffHours × price_per_hour. 
Все це відбувається без оновлення сторінки, завдяки події «input» на полях часу. 
При натисканні «Підтвердити бронювання» виконується AJAX-запит на створення бронювання зі статусом «pending».

В.2. Управління бронюваннями

Вкладка «Мої бронювання» (рис. 2.17) показує таблицю з історією бронювань користувача. 
Статуси відображаються кольоровими бейджами: зелений — «Активне», жовтий — «Очікує», сірий — «Завершено», червоний — «Скасовано». 
Для активних або очікуючих бронювань доступна дія «Скасувати». При скасуванні паркомісце автоматично звільняється.

В.3. Управління транспортними засобами

Вкладка «Мій транспорт» (рис. 2.18) дозволяє керувати особистими автомобілями. Додавання: модальне вікно з полями реєстраційного номера, марки, моделі, кольору та завантаження фото. 
Редагування — з попереднім заповненням форми. Видалення — з підтвердженням SweetAlert2. 
Транспортний засіб, що припаркований на паркінгу, відображає його назву; якщо місце не вибрано — «Не припарковано».

Г. Перевірка адаптивності та сумісності

При зміні ширини вікна браузера інтерфейс коректно перебудовується: десктопний сайдбар (ширина 260px, фіксований) зникає, з'являється мобільний topbar (ширина 100%, з кнопкою гамбургер-меню). 
При натисканні кнопки сайдбар виїжджає зліва (transform: translateX(-100%) → translateX(0)) з overlay-ефектом темного фону. 
Таблиця перетворюється на картковий вигляд на мобільних, KPI-картки розташовуються в колонку. 
Графік Chart.js автоматично масштабується (responsive: true, maintainAspectRatio: false).

Д. Валідація та обробка помилок

Порожні обов'язкові поля форм блокую відправку та підсвічуються червоним кордоном з автоматичним скролом до помилки (form.reportValidity()).
При помилці мережевого з'єднання (відключення інтернету, недоступність сервера) система відображає повідомлення «Проблема зі з'єднанням» через SweetAlert2.
Для всіх операцій з даними використовуються prepared statements (MySQLi), що забезпечує захист від SQL-ін'єкцій. Паролі зберігаються у хешованому вигляді (PASSWORD_DEFAULT).

Е. Повна перевірка бізнес-процесу «від реєстрації до виїзду з паркінгу»

В рамках демонстрації системи було проведено комплексне тестування цільного ланцюга операцій:

1. Адміністратор створює новий паркінг «Аеропорт» (вул. Літакова, 1, місткість 50, тариф 40 грн/год, завантажує зображення). 
   У БД виконується INSERT-запит, паркінг з'являється у таблиці. Доступно місць = 50.

2. Новий користувач реєструється в системі, отримує роль «user».

3. Користувач додає транспорт: Toyota Camry, номер KA 2024 BB, колір сірий.

4. Користувач переходить на вкладку «Паркінги», бачить новий паркінг з вільними 50 місцями і бейджем «49 вільних» (відлік починається з capacity−1).

5. Користувач натискає «Забронювати», обирає Toyota Camry, вказує час заїзду «сьогодні 14:00», виїзду «сьогодні 16:00». 
   Система розраховує: 2 години × 40 грн = 80 грн. Бронювання створюється зі статусом «pending». 
   Доступно місць зменшується до 49.

6. Адміністратор у вкладці «Бронювання» бачить новий запис. Натискає «Підтвердити». 
   Статус змінюється на «active». Бронювання активне.

7. При переході на сторінку після закінчення часу бронювання (end_time < NOW()) система автоматично переводить статус у «completed», доступно місць повертається до 50. 
   Це відбувається без планувальника — при кожному завантаженні сторінки виконується «псевдо-Cron» запит.

8. Адміністратор експортує звіт CSV, відкриває в Excel — кирилиця відображається коректно завдяки BOM-маркеру (0xEF 0xBB 0xBF).

Резюме: система повністю відповідає визначеним у постановці задачі вимогам, забезпечує повний цикл управління парковочними місцями — від реєстрації користувачів та додавання паркінгів до бронювання місць, контролю статусів адміністратором та автоматичного завершення прострочених записів.
