2 Спеціальна частина

2.1 Розробка інформаційної системи управління паркінгом

2.1.1 Постановка задачі
Інформаційна система управління паркінгом призначена для комплексної автоматизації процесів резервування, контролю та обліку паркувальних місць. Програмний продукт забезпечує функціональність для двох основних категорій користувачів: клієнтів (звичайних користувачів) та адміністраторів. Клієнти системи мають змогу створювати облікові записи, проходити процедуру авторизації, додавати власні транспортні засоби із зазначенням реєстраційних номерів та характеристик, а також переглядати перелік доступних паркувальних майданчиків із динамічною індикацією їх зайнятості. Крім того, користувачі можуть здійснювати бронювання місць на визначений час та керувати власними резервуваннями. Адміністратори наділяються розширеними правами доступу, що включають управління даними про паркінги, моніторинг усього зареєстрованого транспорту, управління всіма транзакціями бронювань з можливістю їх ручного підтвердження або відхилення, а також доступ до консолідованих аналітичних даних.

2.1.2 Мета створення інформаційної системи
Головною метою створення системи є забезпечення ефективного та прозорого управління паркувальними зонами шляхом автоматизації ключових бізнес-процесів. Для досягнення цієї мети система вирішує низку інформаційних завдань, а саме: надання точних даних щодо загальної кількості паркувальних місць на кожній локації; визначення актуального співвідношення зайнятих та вільних місць з урахуванням перетину часових інтервалів; ідентифікація транспортних засобів та їхніх власників із зазначенням поточного місця розташування; відстеження історії бронювань, їх статусів та часових меж. Окремим завданням є розробка інтуїтивно зрозумілого, високоінтерактивного веб-інтерфейсу, що функціонує за принципами односторінкового застосунку (SPA), забезпечуючи виконання операцій без необхідності повного перезавантаження сторінки браузера.

2.1.3 Аналіз предметної області
У межах досліджуваної предметної області виділено чотири базові інформаційні сутності. Першою сутністю є користувачі, які мають доступ до системи. Кожен обліковий запис характеризується унікальним ідентифікатором, логіном, хешованим паролем, електронною адресою та визначеною роллю. Другою сутністю є транспортні засоби, що належать користувачам. Для кожного автомобіля зберігаються дані про реєстраційний номер, марку, модель, колір та фотографію. Транспортний засіб може бути логічно прив'язаний до певного паркінгу або мати статус неприпаркованого. Третьою базовою сутністю виступають паркінги, що є фізичними зонами для розміщення автомобілів. Їхні характеристики включають назву, адресу, загальну місткість, кількість поточних вільних місць та погодинний тариф. Четвертою сутністю є бронювання, яке виступає сполучною ланкою між користувачем, його транспортним засобом та обраним паркінгом. Запис про бронювання містить часові межі, розраховану вартість послуги та поточний статус. Структуру взаємовідносин між зазначеними таблицями бази даних наведено на рисунку 2.1.

2.1.4 Проєктування та створення бази даних
З огляду на результати аналізу предметної області, розроблено реляційну базу даних під управлінням системи керування базами даних (СУБД) MySQL. Структура бази даних містить чотири основні таблиці із суворо визначеними зовнішніми ключами. Таблиця «users» призначена для зберігання облікових даних користувачів. Її структура реалізована наступним чином:

CREATE TABLE IF NOT EXISTS users (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(30) NOT NULL,
    password VARCHAR(255) NOT NULL,
    email VARCHAR(50),
    role ENUM('admin', 'user') DEFAULT 'user',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Таблиця «parking_places» зберігає інформацію про паркувальні зони та їхні тарифи:

CREATE TABLE IF NOT EXISTS parking_places (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    address VARCHAR(200) NOT NULL,
    capacity INT(4) NOT NULL,
    available INT(4) NOT NULL,
    price_per_hour DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
    image VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Таблиця «vehicles» використовується для зберігання даних про транспортні засоби клієнтів із прив'язкою до власника та паркінгу:

CREATE TABLE IF NOT EXISTS vehicles (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    license_plate VARCHAR(20) NOT NULL,
    brand VARCHAR(50) NOT NULL,
    model VARCHAR(50) NOT NULL,
    color VARCHAR(30) NOT NULL,
    user_id INT(6) UNSIGNED,
    parking_id INT(6) UNSIGNED,
    image VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (parking_id) REFERENCES parking_places(id)
);

Таблиця «bookings» виконує роль журналу транзакцій резервування:

CREATE TABLE IF NOT EXISTS bookings (
    id INT(6) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id INT(6) UNSIGNED,
    parking_id INT(6) UNSIGNED,
    vehicle_id INT(6) UNSIGNED,
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    total_price DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
    status ENUM('pending', 'active', 'completed', 'cancelled') DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (parking_id) REFERENCES parking_places(id) ON DELETE CASCADE,
    FOREIGN KEY (vehicle_id) REFERENCES vehicles(id) ON DELETE CASCADE
);

2.1.5 Написання SQL запитів до бази даних
Система використовує оптимізований набір SQL-запитів для забезпечення функціонування серверної логіки. Усі маніпуляції з даними виконуються через підготовлені вирази (prepared statements) за допомогою розширення MySQLi. Для отримання повного переліку доступних паркінгів використовується базовий запит вибірки:

SELECT * FROM parking_places ORDER BY id DESC;

Найбільш комплексним є запит для формування історії бронювань, який здійснює внутрішнє об'єднання (JOIN) чотирьох суміжних таблиць:

SELECT b.*, u.username, p.name as parking_name, p.address as parking_address, 
       v.license_plate, v.brand, v.model, v.color 
FROM bookings b 
JOIN users u ON b.user_id = u.id 
JOIN parking_places p ON b.parking_id = p.id 
JOIN vehicles v ON b.vehicle_id = v.id 
WHERE b.user_id = ? ORDER BY b.start_time DESC;

Додавання нових об'єктів виконується за допомогою оператора INSERT INTO. Під час створення нового бронювання система перевіряє часові інтервали та перетин з існуючими запитами:

INSERT INTO bookings (user_id, parking_id, vehicle_id, start_time, end_time, total_price, status) 
VALUES (?, ?, ?, ?, ?, ?, 'pending');

Автоматичне завершення прострочених резервувань та синхронізація реальної місткості паркінгів реалізовано через підзапити агрегації:

UPDATE bookings SET status = 'completed' 
WHERE status = 'active' AND end_time < NOW();

UPDATE parking_places p SET available = capacity - (
    SELECT COUNT(*) FROM bookings b 
    WHERE b.parking_id = p.id AND b.status IN ('pending', 'active') 
    AND b.start_time <= NOW() AND b.end_time > NOW()
);

2.2 Розробка архітектури клієнтської частини та інтерфейсу

2.2.1 Структура DOM-дерева та використання CSS-фреймворку
Клієнтська частина програмного продукту побудована на базі мови розмітки HTML5 з інтенсивним використанням компонентів CSS-фреймворку Bootstrap 5.3. Архітектурно інтерфейс розділено на модульні PHP-шаблони: «header.php» для підключення залежностей та рендерингу навігаційного меню, «footer.php» для завантаження JavaScript-бібліотек, «modals.php» для зберігання розмітки спливаючих вікон та центральний файл «index.php», що відповідає за генерацію контенту. Для створення адаптивної сітки використовуються класи Bootstrap, зокрема «row», «col-md-6», «col-xl-3», що дозволяють динамічно змінювати розташування елементів залежно від ширини екрана пристрою.

Візуальні компоненти, такі як картки паркінгів або статистичні блоки дашборда, побудовані з використанням класу «card» та його похідних («card-header», «card-body»). Форми вводу даних (додавання транспорту, реєстрація) переведені на сучасний стандарт «form-floating», що забезпечує анімацію переміщення підказок (label) при фокусуванні на полі вводу. Усі модальні вікна ініціалізуються через клас «modal fade» і відцентровані на екрані за допомогою атрибута «modal-dialog-centered».

2.2.2 Реалізація парадигми Single Page Application (SPA)
Ключовою технічною особливістю клієнтської частини є відмова від класичної моделі навігації з повним перезавантаженням сторінки. Натомість реалізовано логіку Single Page Application за допомогою асинхронних запитів. У файлі «script.js» розроблено функцію «updateUI()», яка викликається після кожної успішної транзакції (додавання, видалення, оновлення статусу). Ця функція використовує сучасний стандарт Fetch API для виконання GET-запиту до поточного URL. Отриманий HTML-код обробляється за допомогою об'єкта DOMParser, який конвертує текстовий рядок у віртуальне DOM-дерево.

Після парсингу скрипт знаходить блок з ідентифікатором «adminTabsContent» або «userTabsContent» у віртуальному дереві та замінює властивість «innerHTML» відповідного блоку на реальній сторінці. Такий підхід дозволяє оновлювати лише таблиці з даними та графіки, залишаючи стан навігаційного меню недоторканим. Завдяки цьому досягається висока швидкодія інтерфейсу, що імітує роботу нативного мобільного застосунку.

2.2.3 Делегування подій (Event Delegation) та керування формами
Оскільки значна частина DOM-елементів (таблиці, кнопки) постійно перезаписується асинхронною функцією оновлення, пряме навішування обробників подій (через addEventListener на конкретний елемент) призводило б до їх втрати після першого ж оновлення сторінки. Для вирішення цієї архітектурної проблеми застосовано паттерн делегування подій. Єдиний глобальний слухач події «click» навішується на об'єкт «document». Усередині обробника використовується метод «e.target.closest()» для ідентифікації елемента, на який було здійснено натискання.

Наприклад, при натисканні на кнопку з класом «btn-delete-booking», скрипт зчитує атрибут «data-id», ініціює виклик бібліотеки SweetAlert2 для підтвердження дії, а потім надсилає запит на видалення до API. Аналогічний підхід використовується для відкриття модальних вікон редагування: скрипт відправляє запит на отримання поточних даних сутності у форматі JSON, заповнює відповідні поля форми через властивість «value» та програмно викликає об'єкт модального вікна Bootstrap («new bootstrap.Modal(el).show()»).

2.2.4 Динамічні обчислення на стороні клієнта
Окремим програмним модулем клієнтської частини є алгоритм розрахунку вартості бронювання в режимі реального часу. На поля форми з ідентифікаторами «bookingStartTime» та «bookingEndTime» навішено обробник події «input». При кожній зміні значення скрипт виконує нормалізацію часу: перевіряє кількість хвилин та примусово округлює їх до найближчого значення, кратного 30 (00 або 30 хвилин). Якщо користувач обирає час завершення бронювання, що передує часу початку, скрипт автоматично зміщує мінімальний поріг валідації. Після проходження локальної перевірки обчислюється різниця в мілісекундах між двома об'єктами Date, яка конвертується в години та множиться на поточний тариф паркінгу, записаний у глобальну змінну. Розраховане значення миттєво рендериться у відповідному блоці модального вікна.

2.3 Розробка серверної логіки та API

2.3.1 Організація маршрутизації та обробка запитів
Серверна архітектура побудована за принципом єдиної точки входу (Front Controller) для всіх асинхронних операцій через файл «api.php». Маршрутизація реалізована за допомогою конструкції «switch ($action)», де змінна дії отримується з глобального масиву GET. Цей скрипт приймає POST-запити від клієнта (у форматі об'єкта FormData), встановлює HTTP-заголовки «Content-Type: application/json» та повертає результати виконання у вигляді JSON-об'єктів (наприклад, `{"success": true}`). У разі виникнення виняткової ситуації формується JSON-відповідь із ключем «error», яка згодом перехоплюється клієнтським скриптом для виведення сповіщення.

Перед доступом до критичних маршрутів виконується жорстка верифікація сесії: якщо масив «$_SESSION['user_id']» порожній, сервер повертає статус-код 401 (Unauthorized). Окремо перевіряється роль користувача («$is_admin»): якщо клієнт намагається виконати дію «delete_parking» без відповідних прав, виконання скрипта переривається зі статус-кодом 403 (Forbidden).

2.3.2 Реалізація транзакцій та валідація бізнес-логіки
Найскладнішим елементом серверної частини є метод «add_booking». Процес створення резервування обгорнуто в механізм транзакцій бази даних («$conn->begin_transaction()»). Всередині транзакції виконується низка критичних перевірок. По-перше, сервер перевіряє кратність переданого часу (використовується оператор остачі від ділення на 1800 секунд). По-друге, виконується SQL-запит на пошук перетину (overlap check) часових інтервалів для вибраного автомобіля. Якщо транспортний засіб уже має активне бронювання в іншому місці на цей час, транзакція відкочується («$conn->rollback()»). 

Третім рівнем захисту є перевірка ліміту місткості. Виконується запит «SELECT COUNT(*)» для підрахунку кількості існуючих бронювань на обраний паркінг у заданий проміжок часу. Дані про місткість паркінгу блокуються для модифікації іншими потоками за допомогою конструкції «FOR UPDATE». Якщо поточна кількість автомобілів досягає загальної місткості паркінгу, генерується виключення (Exception) і запис до бази не виконується.

2.3.3 Об'єктно-орієнтований підхід до завантаження файлів
Для керування графічними даними (фотографії автомобілів та паркінгів) розроблено окремий PHP-клас «Uploader.php». Такий об'єктно-орієнтований підхід забезпечує ізоляцію коду та можливість його багаторазового використання. Клас інкапсулює логіку перевірки файлу на наявність помилок завантаження («UPLOAD_ERR_OK»), лімітування розміру файлу та строгу верифікацію MIME-типу за допомогою функції «finfo_file()». Остання перевіряє фактичний вміст файлу, а не його розширення, що є важливим заходом безпеки для запобігання завантаженню виконуваних скриптів на сервер. Схвалені файли перейменовуються за допомогою функції «uniqid()» задля уникнення колізій імен та переміщуються в цільову директорію системою.

2.3.4 Впровадження псевдофонового обробника (Псевдо-Cron)
Для підтримки актуальності статусів бронювань розроблено сервісний механізм, що імітує роботу планувальника завдань (Cron). Логіка розміщена на початку головного файлу «index.php». Під час кожного завантаження сторінки або надходження AJAX-запиту від клієнта, сервер ініціює власну транзакцію, яка виконує SQL-інструкцію «UPDATE bookings SET status = 'completed' WHERE status = 'active' AND end_time < NOW()». 

Відразу після цього запускається глобальний запит на перерахунок поля «available» для всіх паркінгів системи. Цей підхід забезпечує консистентність даних без необхідності налаштування демонів на сервері операційної системи. Якщо транзакція стикається з блокуванням бази даних, вона відкочується за допомогою перехоплення виключень (try-catch), що не порушує основний потік рендерингу сторінки користувачеві.
